# Blog post prompt header

This header is prepended to every prompt that creates or edits a blog post on getlina.app. It exists so that individual prompts can focus on the content-specific decisions (slug, i18n keys, sections, concrete anchors) without repeating the boilerplate below.

**How to use:** Paste this entire file into Claude Code, followed by the content-specific section of your prompt. The content section should reference section identifiers (e.g. `blog10.s1title`) and specify the English text for each section — nothing more.

---

## 1. Project context

You are working in the Lina website repo at `/Users/lunde/Developer/lina-website`.

Lina (getlina.app) is a co-parenting app for parents sharing care of a child across two homes. The website targets Nordic and European markets in 8 languages (English root + NO, SV, DA, FI, DE, NL, FR). SEO and professional outreach are the primary growth channels.

## 2. Authoritative style guide

Before writing or editing any content, read `/docs/writing-style.md`. It is authoritative on:
- Tone and voice rules (calm adult tone, no exclamation marks, care vocabulary)
- Required concrete anchors (timeframe, age range, named resource, or specific procedure per section)
- Banned constructions (Not X / but Y, aphoristic endings, em-dash overuse, parallel-symmetric pairs, faux-folksy imperatives, triadic cadence, AI-signature phrases)
- Em-dash budget (maximum 6 per article body, never in headings, never in Not X — Y)
- Translation rules (all rules apply in every language, not only English)

Do not rely on existing articles as tonal templates. Blog posts 1, 4, and 6 are reasonable references. Blog posts 7, 8, and 9 contain slop patterns the style guide explicitly rejects.

## 3. File structure (immutable conventions)

Every blog post has:
- One English source HTML at `/blog/<english-slug>/index.html`
- Per-language variants generated into `/{lang}/blog/<localized-slug>/index.html` for 7 languages
- Localized content driven by `data-i18n="blogN.*"` attributes resolved from `/locales/{lang}.json`
- Matching JSON-LD `Article` and `BreadcrumbList` blocks
- Hreflang with de-AT and de-CH aliases pointing at the German variant

The English source HTML is the only file where content prose lives in the HTML itself (as the English fallback text between the `data-i18n` tags). For every other language, prose lives exclusively in the locale JSON files.

## 4. Build pipeline

All language variants are generated by:

```
node scripts/build-i18n.js
```

The script:
1. Normalises the English source HTML (canonical, hreflang, asset paths, lang-switcher)
2. Generates 7 translated copies using `/locales/{lang}.json`
3. Applies per-language URL slugs from the `langUrls` map in `SOURCE_FILES`
4. Writes `sitemap.xml` with full hreflang support
5. Runs structural safety checks (section count, file size ratio 80–125%) — aborts the affected file on mismatch

Manual edits to `/no/`, `/sv/`, `/da/`, `/fi/`, `/de/`, `/nl/`, or `/fr/` directories are overwritten on the next build. All customisations must live in either the English source HTML, the locale JSON files, or `scripts/build-i18n.js`.

## 5. Registering a new blog post

A new blog post requires three coordinated updates:

**5a. Add to `SOURCE_FILES` in `scripts/build-i18n.js`** (after the last blog entry):

```js
{
  src: 'blog/<english-slug>/index.html',
  urlPath: '/blog/<english-slug>/',
  langUrls: {
    no: '/blog/<norwegian-slug>/',
    sv: '/blog/<swedish-slug>/',
    da: '/blog/<danish-slug>/',
    fi: '/blog/<finnish-slug>/',
    de: '/blog/<german-slug>/',
    nl: '/blog/<dutch-slug>/',
    fr: '/blog/<french-slug>/',
  },
  priority: '0.6',
  hasDropdown: true,
},
```

**5b. Add to `ARTICLE_MAP` in `scripts/build-i18n.js`:**

```js
'blog/<english-slug>/index.html': 'articleN',
```

**5c. Add to `INTERNAL_HREFS` (both with and without trailing slash):**

```js
'/blog/<english-slug>/',
'/blog/<english-slug>',
```

The `INTERNAL_HREFS` list is order-sensitive — longer/specific paths must come before shorter ones to avoid partial overlaps. Place new entries near the existing `/blog/` entries.

## 6. Locale file updates (all 8 files)

Each new blog post requires adding keys to every locale file (`en.json`, `no.json`, `sv.json`, `da.json`, `fi.json`, `de.json`, `nl.json`, `fr.json`):

- A new `blogN` object with keys: `pageTitle`, `pageDesc`, `label`, `title`, `date`, `intro`, `s1title`...`s6title`, `s1p1`...`s6p3`, `ctaTitle`, `ctaDesc`
- Three keys inside the existing `blogIndex` object: `aNlabel`, `aNtitle`, `aNdesc`
- A new `articleN` object inside `jsonLd` with `headline` and `description`

Translation rules — enforce in every language, not only English:
- No exclamation marks anywhere
- Em-dash budget (≤6 per article body; Nordic languages should use fewer by default)
- No "Not X, but Y" / "nicht X, sondern Y" / "pas X, mais Y" / "niet X, maar Y" constructions
- No aphoristic paragraph endings
- Care vocabulary: omsorg/samvær (NO/DA), omsorg/umgänge (SV), huoltajuus/tapaaminen (FI), Umgang/Betreuung (DE), zorg/omgang (NL), garde/coparentalité (FR)
- Match the tonal register of `blog1`, `blog4`, and `blog6` values in the same locale file, not `blog7`, `blog8`, or `blog9`

## 7. Blog index card

Add a new `<a class="blog-card">` entry to `/blog/index.html` with:
- `href="/blog/<english-slug>/"`
- `data-i18n` keys: `blogIndex.aNlabel`, `blogIndex.aNtitle`, `blogIndex.aNdesc`
- English fallback text inside the tags

The new card is typically appended as the last card. Existing cards do not need to be renumbered.

## 8. Related articles block

At the bottom of each article's content, add a "Related articles" section linking to 2–3 related posts. Use the established inline-styled `<section>` and `<ul>` pattern visible in `/blog/handover-day/index.html`. Reference linked posts by their `blogN.title` i18n key so translations flow automatically.

## 9. JSON-LD

Every article has two JSON-LD blocks in `<head>`:

**Article block** — `datePublished`, `dateModified`, `mainEntityOfPage.@id` point at the canonical English URL. The build script localises `headline`, `description`, and `mainEntityOfPage.@id` per language using `jsonLd.articleN` from the locale file.

**BreadcrumbList block** — three items: Home, Blog, and the article title. The build script localises item URLs and the last item's `name` per language.

Do not hand-edit the JSON-LD in generated language files. Only the English source.

## 10. Safety checks and verification

After completing file edits, run:

```
node scripts/build-i18n.js
```

Expected output:
- `Updated (en): blog/<english-slug>/index.html` with matching section count
- 7 language variants generated at the correct localised slugs
- `Generated: sitemap.xml`
- `Done ✓` with 0 errors

If any safety check fails (section count mismatch or size ratio out of 80–125%):
- Stop. Do not continue with further edits.
- Report which file, which language, and the exact mismatch numbers.
- Wait for user instruction before retrying.

## 11. Deployment note

The Cloudflare Worker at `workers/lang-redirect.js` must be deployed manually via the Cloudflare dashboard if it has been changed. A GitHub push does not auto-deploy the Worker. Blog post additions alone do not require Worker changes — but if the prompt also modifies the Worker, this step must not be skipped.

## 12. Things not to do

- Do not edit generated files under `/no/`, `/sv/`, `/da/`, `/fi/`, `/de/`, `/nl/`, or `/fr/`. They will be overwritten.
- Do not add new fields to `LOCALE_SCREENSHOT_MAP` unless the post genuinely has new screenshot assets. The existing keys are load-bearing for the build loop.
- Do not invent Nordic or European family-service resource names in text. Only use verified names (familievernkontoret, familjerätten, Familieretshuset, perheasioiden sovittelu, Jugendamt, Juridisch Loket, Point-justice, Bufdir).
- Do not add exclamation marks in any language.
- Do not reach for em-dashes to achieve a "dramatic pause" effect. See style guide rule 4.
